# 推荐算法设计文档

> **注意：一期推荐逻辑属 B 层（轻量规则引擎），二期将由 LLM 决策取代。**

## 两层架构

### 第一层：硬分档（Tiering）

根据用户当前库存、厨具、食材状态，将所有菜谱分为三档：

| 档位 | 条件 | 说明 |
|------|------|------|
| 🟢 现在就能做 | 所有食材 + 厨具齐全 | 无需任何采购 |
| 🟡 清库存推荐 | 食材齐全，但含久放食材 | 督促尽快消耗，避免浪费 |
| 🔵 需额外购买 | 缺少少量食材/厨具 | 缺的少才推荐，缺太多直接跳过 |

**分档优先级**：食材齐全 + 有久放食材 → 清库存档 > 现在就能做档

### 第二层：档内评分（Scoring）

每档内按加权评分排序，取 Top N（默认 4 道）：

| 维度 | 权重 | 说明 |
|------|------|------|
| 不重样 | 0.35 | 距上次做该菜天数 / 30，从未做过给满分 |
| 清库存 | 0.25 | 含久放食材数量 / 3，仅清库存档参与 |
| 耗时匹配 | 0.20 | 仅用户设置时间筛选时生效 |
| 营养搭配 | 0.20 | 近 3 天连续荤菜 → 给纯素/多蔬菜加分 |

## 清库存阈值

食材处于 `enough` 状态超过 N 天即进入"久放"判定：

| 分类 | 阈值（天） |
|------|-----------|
| 蔬菜 | 3 |
| 肉类 | 7 |
| 蛋奶豆制品 | 5 |
| 主食/调料 | 不提醒 |

## 缺失维度降级策略

筛选器中某维度缺失时优雅降级：
- **烹饪方式**：菜谱无 `method` 属性 → 视为匹配（不过滤）
- **辣度**：菜谱无 `spiciness` → 视为匹配
- **荤素**：菜谱无 `diet_type` → 视为匹配
- **烹饪时间**：菜谱无 `cook_time_minutes` → 视为匹配

## 购物清单闭环

1. 用户勾选想做的菜 → 调用 `generateShoppingList`
2. 清单包含：选中菜谱缺失食材 + 库存不足的调料/主食/蛋奶
3. 用户勾选已采购项 → 调用 `checkoutShoppingList`
4. 对应库存项标记为 `enough`，完成闭环

## 配置

所有参数集中在 `src/lib/recommend/config.ts`，便于二期替换为 LLM 决策。
